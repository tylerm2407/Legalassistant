# ADR 015 — Static Python data for the Know Your Rights library

**Date:** 2026-03-28
**Status:** Accepted
**Deciders:** CaseMate founding team

---

## Decision

The 19 rights guides in the Know Your Rights library are stored as static Python data — a list of `RightsGuide` Pydantic model instances defined directly in `backend/knowledge/rights_library.py`. Guides are not stored in the database, not loaded from files, and not generated by an LLM at runtime.

---

## Context

CaseMate includes a Know Your Rights library that provides pre-written legal guides covering common scenarios across the 10 legal domains. These guides give users immediate, structured information before they even ask a question — eviction defense, security deposit recovery, wage theft, debt collector harassment, and so on.

The guides are reference content, not personalized. They do not change based on the user's profile or state (though they note where state-specific variations exist). The content is curated, reviewed, and static between deployments.

We needed to decide where this content lives: in the database, in external files, or in code.

---

## The implementation

`backend/knowledge/rights_library.py` defines a `RightsGuide` Pydantic model with 10 fields: `id`, `domain`, `title`, `description`, `explanation`, `your_rights` (list[str]), `action_steps` (list[str]), `deadlines` (list[str]), `common_mistakes` (list[str]), and `when_to_get_a_lawyer` (str).

The `RIGHTS_GUIDES` list contains 19 instantiated `RightsGuide` objects covering domains including landlord_tenant, employment_rights, consumer_protection, debt_collections, small_claims, contract_disputes, traffic_violations, family_law, criminal_records, and immigration.

Four accessor functions provide the API layer:

- `get_all_guides()` — Returns the full list
- `get_guide_by_id(guide_id)` — Lookup by unique ID string
- `get_guides_by_domain(domain)` — Filter by legal domain
- `get_domains()` — Returns domain summary with guide counts

These are called by three endpoints in `backend/main.py`: `GET /api/rights/domains`, `GET /api/rights/guides`, and `GET /api/rights/guides/{guide_id}`. All endpoints require authentication but serve the same content to every user.

---

## Alternatives considered

**Database-stored guides in Supabase**
Considered. Would allow updating guide content without code deploys and enable a CMS-like editing experience. Rejected because the guides are reference content that changes infrequently (legal rights do not change weekly), and database storage adds a read query to every guide request. The guides also benefit from version control — changes to legal content should be code-reviewed, not edited in a database admin panel.

**Markdown files with frontmatter**
Considered. Would separate content from code and allow non-developers to edit guides. Rejected because it adds file I/O on every request, requires a markdown parser, and loses Pydantic type validation. The structured fields (your_rights, action_steps, deadlines, common_mistakes) are better represented as typed lists than as markdown sections that need parsing.

**LLM-generated guides at runtime**
Rejected. Would add latency and cost to every guide view, produce inconsistent content between requests, and risk generating inaccurate legal information. Static, human-reviewed content is safer for legal reference material.

**External CMS (Contentful, Strapi)**
Rejected. Adds an external dependency, API calls, and infrastructure for 19 documents that change rarely. Over-engineered for the current content volume.

---

## Consequences

**Positive:**
- Zero latency — guides are in-memory Python objects, no I/O required
- Type-safe — every guide is validated by Pydantic at import time
- Version-controlled — changes go through git, enabling code review for legal content
- No additional infrastructure — no CMS, no external API, no database table
- Testable — guide structure and content can be verified in unit tests

**Negative:**
- Content updates require a code deploy, even for typo fixes
- Non-developers cannot edit guide content without touching Python code
- 19 guides in a single file creates a large module (the file is substantial in size)
- No internationalization support — guides are English-only
- No per-state customization of guide content (guides are generic across states)

---

## Status

Accepted. Static Python data is the right choice for 19 curated reference guides. If the guide count grows past 50 or non-developer editing becomes a requirement, migrating to database storage with a simple admin interface would be a straightforward refactor — the `RightsGuide` model and accessor functions would remain unchanged, only the data source would change.
