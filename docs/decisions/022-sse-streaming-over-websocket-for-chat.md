# ADR 022 — SSE Streaming over WebSocket for Chat

**Status:** Accepted
**Date:** 2026-03-29
**Decision Makers:** Tyler Moore, Owen Ash

---

## Context

CaseMate's chat responses are generated by the Anthropic API, which streams tokens incrementally. The user should see tokens appear in real time as Claude generates them — not wait 3-8 seconds for a complete response to load all at once. Streaming is a UX requirement, not an optimization. Users perceive streaming responses as faster even when total generation time is identical.

The communication pattern is unidirectional: the server sends tokens to the client as they arrive from the Anthropic API. The client does not need to send data back to the server during a response stream — the user's message was already sent via `POST /api/chat`, and the stream is a follow-up read on the response.

We evaluated three real-time delivery mechanisms: Server-Sent Events (SSE), WebSockets, and long polling. The decision needed to account for: infrastructure complexity (we are a 2-person team), compatibility with CDNs and corporate proxies (our users are regular people, not developers — they browse from work networks, coffee shop WiFi, mobile carriers), and the FastAPI backend we already have.

---

## Decision

Use Server-Sent Events (SSE) via `GET /api/chat/{id}/stream` for real-time token delivery.

The implementation:

1. Client sends the user's message via `POST /api/chat`, which returns a `chat_id` and begins background processing.
2. Client opens an SSE connection to `GET /api/chat/{id}/stream` using the native `EventSource` API (or a polyfill for additional control).
3. Backend uses FastAPI's `StreamingResponse` with `media_type="text/event-stream"` to push `ChatChunk` events as tokens arrive from the Anthropic API.
4. Each SSE event contains a JSON payload: `{"type": "token", "content": "..."}` for text chunks, `{"type": "done", "usage": {...}}` for stream completion, and `{"type": "error", "message": "..."}` for failures.
5. The connection closes automatically when the stream completes. No persistent connection management needed.

SSE events are plain HTTP responses. They work through every CDN, reverse proxy, load balancer, and corporate firewall without special configuration. There is no upgrade negotiation, no special port requirements, and no connection state to manage on the server.

---

## Alternatives Considered

### WebSocket

WebSocket provides full-duplex, bidirectional communication over a single TCP connection. It is the standard choice for real-time applications where both sides send data — chat apps, collaborative editors, multiplayer games.

CaseMate does not need bidirectional streaming. The user sends a message (one HTTP POST), and the server streams back a response. There is no scenario where the client needs to send data to the server while a response is being generated. WebSocket is overqualified for this use case, and that overqualification has real costs:

- **Proxy compatibility:** Many corporate proxies, hotel WiFi captive portals, and mobile carrier proxies do not support WebSocket upgrade negotiation. They silently drop the connection or return 400 errors. Our users are regular people — not developers who can debug network issues. SSE works through all of these because it is plain HTTP.
- **Server complexity:** WebSocket requires managing connection state — tracking which connections are alive, handling disconnects, implementing heartbeats. FastAPI supports WebSocket but it adds a different programming model (async generators with connection lifecycle management) compared to a simple StreamingResponse.
- **Load balancer configuration:** WebSocket connections require sticky sessions or special load balancer configuration to maintain the connection to the same backend instance. SSE connections are stateless HTTP requests that work with any load balancer out of the box.
- **Reconnection:** SSE has built-in automatic reconnection with `Last-Event-ID` support. WebSocket reconnection must be implemented manually.

Rejected. The bidirectional capability is not needed, and the infrastructure cost of supporting it is not justified.

### Long Polling

The client repeatedly sends HTTP requests asking "do you have more tokens?" The server holds the request open until a token is available, then responds and the client immediately sends another request.

- **Latency:** Each token requires a full HTTP request/response cycle. Even with keep-alive connections, the overhead per token is significant compared to a persistent stream. Users would see choppy, stuttering text instead of smooth streaming.
- **Server resources:** Each poll is a separate request that consumes a server thread/connection. At scale, this means 10-50x more requests than SSE for the same response.
- **Complexity:** Requires server-side buffering of tokens between polls, tracking which tokens have been delivered, and handling out-of-order or duplicate polls.

Rejected. Long polling is a compatibility fallback for environments that cannot support SSE, which is effectively zero modern browsers.

### gRPC Server Streaming

gRPC supports server-side streaming natively and would provide strongly typed message contracts via protobuf. However:

- **Browser support:** gRPC requires gRPC-Web, a separate proxy layer that translates between browser HTTP and gRPC's HTTP/2 framing. This adds infrastructure complexity.
- **Protobuf overhead:** Defining .proto files for chat chunks is over-engineering for a simple `{type, content}` payload.
- **Team familiarity:** Neither team member has production gRPC experience. SSE is trivial to implement and debug.

Rejected. gRPC streaming is the right choice for service-to-service communication, not browser-to-server token delivery.

## Consequences

### Positive

- Zero infrastructure changes — SSE works with FastAPI's built-in `StreamingResponse`, our existing Nginx/Caddy reverse proxy, and any CDN
- Native browser support — `EventSource` API is supported in all modern browsers with no dependencies
- Automatic reconnection — if the connection drops mid-stream, the browser reconnects automatically and can resume from `Last-Event-ID`
- Simple server implementation — a single async generator function that yields tokens from the Anthropic stream, wrapped in `StreamingResponse`
- HTTP/2 multiplexing — multiple SSE connections share a single TCP connection, so the browser's 6-connection-per-domain limit (HTTP/1.1) is not a concern
- Easy debugging — SSE events are plain text, visible in browser DevTools Network tab without special tooling

### Negative

- Unidirectional only — if we ever need client-to-server streaming (e.g., real-time voice input), SSE cannot handle it and we would need to add WebSocket alongside it
- No binary data — SSE is text-only. If we ever stream binary content (audio, images), we would need a different mechanism. Not relevant for text chat.
- Connection limit on HTTP/1.1 — browsers limit to 6 connections per domain on HTTP/1.1. If a user opens 7+ CaseMate tabs, the 7th SSE connection would queue. Mitigated by HTTP/2 (which we use) where connections are multiplexed.

### Risks

- If we add features requiring bidirectional real-time communication (collaborative editing, live cursor presence), we will need to introduce WebSocket as a second transport. This is a future concern, not a current one. The two transports can coexist.
- Some older corporate proxy configurations may buffer SSE responses, causing tokens to arrive in batches instead of individually. Mitigation: set `Cache-Control: no-cache` and `X-Accel-Buffering: no` headers to disable proxy buffering.

## Metrics

| Metric | Target | Measurement |
|--------|--------|-------------|
| Time to first token | <200ms from stream open | Client-side timestamp delta: SSE connection opened to first `token` event received |
| Stream completion time | Within 5% of non-streaming equivalent | Compare SSE total time to synchronous `/api/chat` response time |
| Dropped connections | <0.1% of streams | Count of streams that error before receiving a `done` event |
| Reconnection rate | <1% of streams | Count of `Last-Event-ID` reconnections / total streams |
| Proxy compatibility | 100% of tested networks | Test from corporate proxy, mobile carrier, and public WiFi |
